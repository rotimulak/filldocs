# Эвристики и стратегии обработки документов (v2)

## Общий подход v2

В v2 основная логика извлечения и маппинга переложена на LLM. Эвристики применяются только для:
1. Поиска таблицы-кандидата для заполнения
2. Конвертации docx в текстовое представление для LLM

---

## Поиск таблицы реквизитов

### Стратегия (`find_requisites_table`)

1. Перебрать все таблицы документа
2. Для каждой таблицы подсчитать совпадения ключевых слов из LABEL_MAPPING
3. Фильтр: таблица должна иметь ≥2 колонки
4. Порог: ≥2 совпадения (в v1 был ≥3, снижен т.к. LLM компенсирует)
5. Выбрать таблицу с максимальным score

### Ключевые слова для скоринга

Используются паттерны из `docx_filler.py:DEFAULT_LABEL_MAPPING` и `COMPOSITE_FIELDS`:
- наименование, фирменное наименование
- инн, кпп, огрн
- адрес, место нахождения
- банковские реквизиты, бик, расчётный счёт
- телефон, email
- руководитель, директор
- и другие (~50 паттернов)

### Fallback

Если `find_requisites_table` не нашла подходящую таблицу (score < 2):
- Берём первую таблицу из `docx_tables_to_text()`
- Если таблиц нет вообще → HTTP 400

## Конвертация docx → текст

### Параграфы
- Извлекается `paragraph.text`
- Пустые параграфы пропускаются
- Разделитель: двойной перенос строки

### Таблицы → Markdown

```
| ячейка_0_0 | ячейка_0_1 | ячейка_0_2 |
| --- | --- | --- |
| ячейка_1_0 | ячейка_1_1 | ячейка_1_2 |
```

- Переносы внутри ячейки заменяются на пробел
- Разделитель `| --- |` после первой строки (заголовок)
- Количество колонок определяется по максимуму среди строк

### Сохранение порядка
- Итерация по `doc.element.body` children
- Параграфы и таблицы чередуются в порядке появления в документе

## Что делает LLM (а не эвристики)

### При извлечении (extract)
- Понимание контекста полей ("Адрес" → юридический vs почтовый)
- Разделение составных полей ("ИНН/КПП" → два отдельных)
- Извлечение чистых значений из "грязного" текста
- Нормализация числовых полей (только цифры)
- Определение названий полей на русском

### При заполнении (fill)
- Сопоставление реквизитов с ячейками таблицы
- Определение, какие ячейки пустые и предназначены для заполнения
- Генерация точных координат (row, col) для каждого значения
- Учёт структуры таблицы (заголовки, нумерация, метки)

## Обработка ошибок LLM

| Ситуация | Стратегия |
|----------|-----------|
| Невалидный XML | ValueError → HTTP 502 |
| Невалидный JSON | ValueError → HTTP 502 |
| Таймаут API | Retry 3 раза с backoff → HTTP 502 |
| Rate limit (429) | Retry с backoff |
| Server error (5xx) | Retry с backoff |
| Client error (4xx, не 429) | Не retry, сразу ошибка |
| Connection error | Retry с backoff → HTTP 502 |
| Пустой результат | success=false, предупреждение |

### Retry-стратегия
- Максимум: 3 попытки
- Начальный backoff: 1 секунда
- Множитель: 2x (1s → 2s → 4s)
