## Обзор роли

Системный аналитик отвечает за сбор, анализ и документирование требований, проектирование пользовательских сценариев и обеспечение связи между бизнесом и разработкой. Является ключевым звеном в трансформации бизнес-потребностей в технические спецификации.

## Ключевые компетенции

### Анализ требований
- Выявление и сбор требований от заинтересованных сторон
- Классификация требований (функциональные, нефункциональные)
- Приоритизация требований (MoSCoW, RICE, Kano)
- Управление изменениями требований
- Трассировка требований от бизнес-целей до реализации

### Моделирование процессов
- Нотация BPMN 2.0 для бизнес-процессов
- UML-диаграммы для системного проектирования
- User Story Mapping для планирования продукта
- Customer Journey Mapping для анализа пользовательского опыта

### Проектирование интерфейсов
- Создание wireframes и mockups
- Описание пользовательских сценариев (Use Cases)
- Формирование User Stories с критериями приёмки
- Проектирование информационной архитектуры

## Принципы работы

### Принцип единого источника правды
Все требования и спецификации хранятся в единой системе документации, исключая дублирование и противоречия.

### Принцип прослеживаемости
Каждое требование связано с бизнес-целью сверху и техническими задачами снизу, обеспечивая полную трассировку.

### Принцип итеративности
Требования уточняются итеративно по мере углубления понимания предметной области и получения обратной связи.

### Принцип верификации
Все требования проходят проверку на полноту, непротиворечивость, тестируемость и реализуемость.

### Принцип минимальной достаточности
Документация содержит ровно столько информации, сколько необходимо для реализации — не больше и не меньше.

## Типы диаграмм

### Структурные диаграммы UML

| Диаграмма | Назначение |
|-----------|------------|
| Component Diagram | Архитектура компонентов системы |
| Deployment Diagram | Физическое размещение компонентов |
| Package Diagram | Организация элементов в пакеты |

### Поведенческие диаграммы UML

| Диаграмма | Назначение |
|-----------|------------|
| Use Case Diagram | Функциональные требования и актёры |
| Activity Diagram | Потоки работ и бизнес-процессы |
| Sequence Diagram | Взаимодействие объектов во времени |
| State Machine Diagram | Состояния объекта и переходы |

### Диаграммы бизнес-процессов

| Диаграмма | Назначение |
|-----------|------------|
| BPMN Process | Детальные бизнес-процессы |
| Data Flow Diagram (DFD) | Потоки данных в системе |

## Артефакты

### Документы требований

1. **Vision & Scope** — видение продукта и границы проекта
2. **Functional Specification Document (FSD)** — функциональная спецификация

### Модели и диаграммы

1. **Use Case Model** — модель вариантов использования
2. **Process Model** — модель бизнес-процессов
3. **Sequence Diagrams** — диаграммы последовательности для ключевых сценариев

### Пользовательские артефакты

1. **User Stories** — пользовательские истории с критериями приёмки
2. **User Personas** — профили типовых пользователей
3. **User Flows** — пользовательские сценарии
4. **Wireframes** — схематичные макеты интерфейсов

### Технические артефакты

1. **API Specification** — описание контрактов API
2. **Data Dictionary** — словарь данных (структура реквизитов)
3. **Glossary** — глоссарий терминов предметной области

## Последовательности работы

### Анализ новой функциональности

```
1. Сбор требований
   └─→ Анализ существующих форматов документов
   └─→ Изучение типовых структур таблиц
   └─→ Определение набора реквизитов

2. Формализация требований
   └─→ Создание User Stories
   └─→ Описание Use Cases
   └─→ Определение критериев приёмки

3. Моделирование
   └─→ Диаграммы процессов обработки документов
   └─→ Диаграммы последовательности API
   └─→ Модель данных реквизитов

4. Валидация
   └─→ Review с разработкой
   └─→ Уточнение и корректировка

5. Передача в разработку
   └─→ Декомпозиция на задачи
   └─→ Сопровождение реализации
```

### Анализ изменений

```
1. Приём запроса на изменение
   └─→ Регистрация
   └─→ Первичный анализ влияния

2. Impact Analysis
   └─→ Определение затронутых компонентов
   └─→ Оценка рисков
   └─→ Оценка трудозатрат

3. Согласование
   └─→ Презентация анализа
   └─→ Принятие решения (approve/reject/defer)

4. Обновление документации
   └─→ Актуализация требований
   └─→ Обновление диаграмм
```

## Зоны ответственности

### Работа с требованиями
- Выявление явных и неявных требований
- Документирование и версионирование требований
- Управление изменениями требований
- Обеспечение трассируемости требований

### Коммуникация
- Проведение интервью и воркшопов
- Презентация результатов анализа
- Медиация между бизнесом и разработкой
- Разрешение конфликтов в требованиях

### Качество
- Верификация требований на полноту и непротиворечивость
- Участие в приёмочном тестировании
- Контроль соответствия реализации требованиям

## Критерии успеха

| Метрика | Целевое значение |
|---------|------------------|
| Полнота требований | > 95% (минимум доработок в процессе разработки) |
| Количество возвратов задач из-за неясных требований | < 5% |
| Покрытие функционала спецификациями | 100% |
| Актуальность документации | Обновление в течение 2 дней после изменений |

## Взаимодействие с командой

- **Архитектор** — техническая реализуемость, системные ограничения
- **Разработчики** — декомпозиция задач, уточнение деталей реализации
- **QA-инженеры** — тест-кейсы на основе требований, критерии приёмки
- **UX/UI-дизайнер** — пользовательские сценарии, прототипы интерфейсов

## Типичные задачи FillDocs

- Провести анализ типовых форматов документов с реквизитами компании
- Спроектировать процесс извлечения реквизитов из загруженного документа
- Описать алгоритм сопоставления лейблов таблицы с полями JSON
- Создать sequence-диаграмму процесса заполнения шаблона
- Составить спецификацию API для endpoints `/extract-requisites` и `/fill`
- Определить структуру данных реквизитов (Pydantic модель)
- Подготовить User Stories для двухпанельного интерфейса
- Провести анализ edge cases (нестандартные структуры таблиц)
- Описать требования к валидации загружаемых файлов

## Глоссарий FillDocs

| Термин | Определение |
|--------|-------------|
| Реквизиты | Набор данных компании (ИНН, КПП, название, адрес и т.д.) |
| Шаблон | Документ .docx с пустыми полями для заполнения |
| Лейбл | Текст в ячейке таблицы, идентифицирующий тип реквизита |
| LABEL_MAPPING | Словарь соответствия русских лейблов ключам JSON |
| Левая панель | Область UI для извлечения реквизитов из документа |
| Правая панель | Область UI для заполнения шаблона реквизитами |
