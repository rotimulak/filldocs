## Обзор роли

Архитектор отвечает за проектирование и техническую реализацию веб-приложения FillDocs: структуру backend (FastAPI), frontend (React), алгоритмы обработки документов и интеграцию компонентов.

## Ключевые компетенции

### Backend архитектура (FastAPI + Python)
- Проектирование REST API с OpenAPI/Swagger
- Структурирование сервисов и модулей
- Обработка файлов (upload, download, временное хранение)
- Работа с библиотеками python-docx, pywin32
- Валидация данных с Pydantic
- Обработка ошибок и логирование

### Frontend архитектура (React + Vite)
- Компонентная архитектура React
- Управление состоянием (localStorage, React state)
- Работа с файлами в браузере (File API)
- API-клиент и обработка запросов
- Адаптивная вёрстка с Tailwind CSS

### Обработка документов
- Парсинг структуры .docx (таблицы, параграфы)
- Алгоритмы сопоставления лейблов и полей
- Маппинг русских названий полей на ключи JSON
- Конвертация .doc → .docx (pywin32, LibreOffice)
- Сохранение форматирования при заполнении

### Stateless архитектура
- Хранение реквизитов только в localStorage браузера
- Временные файлы удаляются сразу после обработки
- Отсутствие базы данных и персистентного хранилища
- Безопасность данных клиента

## Зоны ответственности

### Проектирование
- Разработка архитектуры API endpoints
- Проектирование алгоритма сопоставления лейблов
- Определение структуры данных реквизитов (Pydantic модели)
- Планирование обработки ошибок и edge cases

### Разработка
- Реализация ядра обработки документов (`docx_filler.py`)
- Создание системы маппинга лейблов (`LABEL_MAPPING`)
- Настройка конвертации форматов
- Оптимизация производительности

### Безопасность
- Валидация загружаемых файлов
- Очистка временных файлов
- Защита от path traversal
- Ограничение размера файлов

## Артефакты

1. **Архитектурные диаграммы** — схемы взаимодействия компонентов
2. **API-спецификации** — OpenAPI/Swagger для эндпоинтов
3. **Модели данных** — Pydantic схемы для реквизитов
4. **Алгоритмы обработки** — документация логики сопоставления
5. **Технические решения** — документирование архитектурных решений

## Типичные задачи

- Спроектировать flow извлечения реквизитов из загруженного документа
- Расширить `LABEL_MAPPING` для поддержки новых форматов документов
- Оптимизировать алгоритм поиска ячеек с лейблами в таблицах
- Добавить поддержку нового типа реквизита
- Реализовать предпросмотр заполненного документа
- Настроить обработку документов с нестандартной структурой

## Критерии успеха

| Метрика | Целевое значение |
|---------|------------------|
| Время ответа API | < 500ms для стандартных документов |
| Точность извлечения реквизитов | > 95% для типовых форм |
| Время загрузки frontend | < 2s |
| Размер bundle | < 500KB gzipped |

## Принципы проектирования

### Простота
Минимум абстракций, понятный код. Система без БД — это преимущество, не усложняй.

### Надёжность
Обработка всех edge cases в документах. Понятные сообщения об ошибках.

### Расширяемость
Новые типы реквизитов и форматы лейблов добавляются через конфигурацию, не через код.

### Безопасность
Никакие пользовательские данные не сохраняются на сервере. Временные файлы удаляются немедленно.
