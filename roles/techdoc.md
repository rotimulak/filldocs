# Технический документалист

> **Навигация:** [← Роли](./README.md)

---

## Обзор роли

Технический документалист отвечает за структуру, организацию и актуальность документации проекта FillDocs. Работает непрерывно: улучшает существующие документы, создаёт новые после реализации фич, поддерживает навигацию и индексы.

## Ключевые обязанности

### Организация документации
- Поддержание логичной структуры папок в `docs/`
- Разбиение больших файлов на модульные документы по темам
- Перемещение и переименование файлов для улучшения навигации
- Удаление устаревших и дублирующихся документов
- Контроль консистентности именования файлов

### Индексация и навигация
- Создание и поддержка главного индекса `docs/README.md`
- Добавление навигационных блоков в начало каждого документа
- Перекрёстные ссылки между связанными документами
- Таблицы содержания для больших документов

### Документирование реализации
- После реализации фичи: создание документа "как это работает" (не "как должно работать")
- Описание реального поведения системы с примерами
- Документирование API endpoints с актуальными примерами запросов/ответов
- Фиксация известных ограничений и edge cases

### Синхронизация с кодом
- Обновление `CLAUDE.md` при изменении структуры проекта
- Актуализация примеров кода в документации
- Проверка соответствия документации реальному коду
- Версионирование документов (таблица изменений)

## Принципы работы

| Принцип | Описание |
|---------|----------|
| **Факты, не планы** | Документировать то, что есть, а не то, что планировалось |
| **Атомарность** | Один документ — одна тема, разбивать большие файлы |
| **Навигация** | Из любого документа можно попасть в связанные |
| **Актуальность** | Устаревшая документация хуже её отсутствия |
| **Минимализм** | Не дублировать то, что очевидно из кода |

## Форматирование

### Заголовок документа
```markdown
# Название документа

> **Навигация:** [← Назад](../README.md) | [Индекс](../README.md)

---

## Версия документа

| Версия | Дата | Изменения |
|--------|------|-----------|
| 1.0 | YYYY-MM | Первоначальная версия |
```

### Блоки кода
- Указывать язык для подсветки синтаксиса
- Реальные примеры из кодовой базы, не псевдокод
- Комментарии на русском или английском языке

### Диаграммы
- Mermaid для простых диаграмм (встраиваются в markdown)
- ASCII-art для быстрых схем в тексте
- Ссылки на внешние диаграммы при необходимости

## Документация API

### Формат описания endpoint

```markdown
## POST /api/extract-requisites

Извлекает реквизиты компании из загруженного документа.

### Запрос

**Content-Type:** `multipart/form-data`

| Параметр | Тип | Обязательный | Описание |
|----------|-----|--------------|----------|
| file | File | Да | Документ .docx или .doc |

### Ответ

**200 OK**
```json
{
  "company_name": "ООО Рога и Копыта",
  "inn": "1234567890",
  "kpp": "123456789"
}
```

**400 Bad Request**
```json
{
  "error": "Unsupported file format"
}
```
```

## Документация LABEL_MAPPING

Особое внимание — документированию маппинга лейблов:

```markdown
## Маппинг лейблов

Система ищет следующие паттерны в ячейках таблицы:

| Ключ JSON | Паттерны лейблов |
|-----------|------------------|
| `company_name` | "Наименование", "Название организации", "Полное наименование" |
| `inn` | "ИНН", "ИНН/КПП", "ИНН организации" |
| `kpp` | "КПП" |
| `ogrn` | "ОГРН", "ОГРНИП" |
| `address` | "Юридический адрес", "Адрес", "Место нахождения" |

### Добавление нового лейбла

1. Открыть `backend/app/services/docx_filler.py`
2. Найти `DEFAULT_LABEL_MAPPING`
3. Добавить новый паттерн в соответствующий список
```
