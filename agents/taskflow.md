# Workflow агентов FillDocs

Этот файл — операционный процесс разработки. Подключается к сессии через тег `@agents/taskflow.md`. После подключения основная сессия работает как orchestrator и ведёт задачу по процессу от начала до конца.

---

## Как запустить

Пользователь в сессии Claude Code описывает задачу. Возможны два входа:

| Вход | Что делать |
|------|-----------|
| **Идея / проблема / запрос** | → Фаза 1 (Research) |
| **Конкретный тикет** (номер Linear или чёткое ТЗ) | → Фаза 3 (Code) |

---

## Процесс

### Фаза 1 — Research

**Кто:** `researcher` (через Task subagent)
**Вход:** описание задачи от пользователя
**Выход:** PRD в `docs/features/{slug}.md` или `docs/discovery/{slug}.md`

Шаги:
1. Уточнить scope у пользователя (что входит, что нет)
2. Делегировать → `researcher` с описанием задачи
3. Кратко показать саммари PRD пользователю
4. Если researcher отметил открытые вопросы — **стоп, спросить пользователя**
5. Если всё штатно → двигаться дальше

### Фаза 2 — Plan

**Кто:** `planner` (через Task subagent)
**Вход:** одобренный PRD + текущие тикеты из Linear
**Выход:** список тикетов для создания в Linear

Шаги:
1. Получить существующие тикеты из Linear (проверка дублей)
2. Делегировать → `planner` с PRD и списком тикетов
3. Если planner нашёл дубли или конфликты → **стоп, показать пользователю**
4. Если всё чисто → создать тикеты в Linear, сообщить пользователю

> Если задача пришла как готовый тикет — фазы 1–2 пропускаются.

### Фаза 3 — Code

**Кто:** `coder` (через Task subagent), опционально `architect`
**Вход:** тикет (ID, описание, критерии приёмки)
**Выход:** коммит в feature-ветку (код + тесты + CHANGELOG + docs)

Шаги:
1. Если тикет затрагивает архитектуру → делегировать → `architect`
   - CRITICAL → **стоп, показать пользователю**
   - WARNING/NOTE → учесть в задаче для coder
2. Делегировать → `coder` с деталями тикета
3. Проверить Definition of Done:
   - [ ] Реализация соответствует критериям приёмки
   - [ ] Тесты написаны
   - [ ] CHANGELOG обновлён
   - [ ] Документация обновлена (если менялась архитектура)
4. DoD выполнен → дальше. Не выполнен → **стоп, доложить пользователю**

### Фаза 4 — Test

**Кто:** `tester` (через Task subagent)
**Вход:** реализованный код + критерии приёмки
**Выход:** отчёт о тестировании

Шаги:
1. Делегировать → `tester` с тикетом и изменёнными файлами
2. Все тесты зелёные → дальше
3. Есть падения → вернуть coder на фикс (без вопросов пользователю)
4. Если после 2 попыток не починены → **стоп, доложить пользователю**

### Фаза 5 — Review

**Кто:** `reviewer` (через Task subagent)
**Вход:** завершённый тикет, изменённые файлы
**Выход:** findings в `docs/reviews/{YYYY-MM-DD}-{ticket-id}-review.md`

Шаги:
1. Делегировать → `reviewer` с тикетом и файлами
2. Critical → вернуть coder (макс. 2 итерации). После 2-й → **стоп, эскалация**
3. Medium/Low → принять, создать follow-up через `planner`
4. Чисто → дальше

### Фаза 6 — Ship

**Кто:** основная сессия (orchestrator)
**Вход:** чистое ревью, все тесты зелёные
**Выход:** PR в GitHub, статус в Linear

Шаги:
1. `git push` + создать PR через `gh`
2. Обновить статус тикета в Linear → Done / In Review
3. Сообщить пользователю: ссылка на PR, что сделано, follow-up (если есть)

### Фаза 7 — Deploy (по запросу)

**Кто:** основная сессия (orchestrator)
**Вход:** замерженный PR в main
**Выход:** обновлённый прод

**ВАЖНО:** Деплой ТОЛЬКО через git. Никогда не копировать файлы на VPS через scp/rsync напрямую — это ломает git-состояние и приводит к потере изменений (stash, staged conflicts).

Шаги:
1. Мерж PR в main: `gh pr merge <N> --squash --delete-branch`
2. SSH на VPS, pull и rebuild:
```bash
ssh -p 443 root@153.80.251.203 'cd /opt/filldocs && git pull origin main && docker compose build --no-cache frontend && docker compose up -d frontend'
```
3. Если git pull конфликтует с untracked файлами — удалить их, pull заменит на свои
4. Если изменился nginx конфиг:
```bash
ssh -p 443 root@153.80.251.203 'cp /opt/filldocs/deploy/nginx-host.conf /etc/nginx/sites-available/filldocs && nginx -t && systemctl reload nginx'
```
5. Проверить результат: `curl -s https://filldocs.ru/ | grep "<title>"`
6. Сообщить пользователю: URL + что проверить

## Правила процесса

1. **Автономность по умолчанию.** Если всё штатно — двигаться вперёд без вопросов. Останавливаться только при проблемах.
2. **Деплой ТОЛЬКО через git pull.** НИКОГДА не копировать файлы на VPS через scp/rsync — это создаёт рассинхрон между git и файловой системой, ведёт к потерянным изменениям и stash-конфликтам. Единственный путь: merge PR → git pull на VPS → docker rebuild.
3. **Когда останавливаться и спрашивать пользователя:**
   - Researcher нашёл открытые вопросы / неоднозначности
   - Planner обнаружил дубли или конфликты
   - Architect выдал CRITICAL
   - Coder не может выполнить DoD
   - Тесты не чинятся после 2 попыток
   - Critical findings в ревью после 2 итераций
4. **Макс. 2 итерации** на фикс тестов и на фикс ревью-findings. Потом — эскалация.
5. **Один тикет за раз.** Довести до конца, потом следующий.
6. **Всё в одном коммите.** Код + тесты + CHANGELOG + docs — один атомарный коммит.
7. **Информировать, не спрашивать.** Краткое саммари после каждой фазы.

## Быстрый путь (тикет напрямую)

```
Пользователь: «Сделай X» (чёткое ТЗ)
    │
    ▼
Фаза 3 (Code) → Фаза 4 (Test) → Фаза 5 (Review) → Фаза 6 (Ship) → Фаза 7 (Deploy)
```

```
Пользователь: «Хочу чтобы ...»
    │
    ▼
Фаза 1 (Research) → Фаза 2 (Plan) → Фаза 3 (Code) → Фаза 4 (Test) → Фаза 5 (Review) → Фаза 6 (Ship) → Фаза 7 (Deploy)
```
